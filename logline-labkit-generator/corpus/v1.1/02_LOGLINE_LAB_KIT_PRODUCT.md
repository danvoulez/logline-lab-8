# 02_LOGLINE_LAB_KIT_PRODUCT.md

## LogLine Lab Kit v0

## Product Thesis

LogLine Lab Kit v0 é a primeira forma pública de produto da ideia LogLine Lab: um kit de laboratório vivo para estudar LogLine em movimento e construir com LogLine sob condições reais.

O produto dá a um operador um lugar de trabalho para carregar material LogLine, declarar prática, nomear runtimes, emitir atos, rodar probes, projetar mundos, preservar ghosts, preparar receipts e produzir reports sobre o que foi estudado ou construído.

O produto não é um conjunto de documentos passivo. É uma superfície de laboratório em funcionamento. Seu valor é a experiência de ver LogLine mover-se através de atos, engines, runtimes, projections, ghosts, probes, evidence, receipts, intervals, clocks, standards, contracts, AI e realidade.

## Product Sentence

```txt
Study LogLine itself.
Run it.
Project it.
Doubt it.
Prove what can be proved.
Preserve what cannot.
```

## Product Identity

### Name

```txt
LogLine Lab Kit v0
```

### Position

```txt
LogLine Foundation is the grammar.
LogLine Lab is the practice kit.
A local lab is a declared instantiation.
```

### Public Promise

Um operador capaz pode instalar o kit e criar um LogLine Lab pequeno que pode:

```txt
load LogLine references
load or declare practice
name runtimes or human procedures
emit or preserve LogLine Acts
keep acts separate from projections
run probes
record evidence
preserve ghosts
prepare scoped receipts
produce at least one Lab report
```

Este é o produto mínimo credível. Qualquer coisa além disso aumenta poder, não legitimidade.

## Product Experience

O produto abre como laboratório, não como repositório.

A primeira experiência é um ciclo Lab guiado:

```txt
Load
Declare
Observe
Emit
Project
Learn
```

O operador deve terminar a primeira sessão com cinco coisas em mãos:

```txt
1. a declared Lab manifest
2. a first LogLine Act or candidate act
3. a first projection from act-shaped material
4. a first ghost naming what cannot yet close
5. a first Lab report describing what was studied or built
```

O produto ensina pelo uso. Não faz o operador ler uma doutrina antes de tocar no Lab. Dá ao operador uma sequência que torna a doutrina visível através da ação.

### First Session

A primeira sessão deve ser curta, séria e completa:

```txt
open the kit
read the product sentence
create or inspect a Lab manifest
load references and practice material
emit or preserve an act
project the act into a readable view
open a ghost for unresolved state
attach or reference evidence if available
prepare a receipt candidate only within scope
write the first Lab report
```

O output da primeira sessão não é um banner de sucesso. O output é um report que mostra o que aconteceu, o que foi observado, o que permanece ghosted e o que pode ser tentado em seguida.

## Operating Model

O Lab opera através de seis movimentos.

### Load

Load traz material para o Lab sem transformá-lo em verdade.

O Lab pode carregar:

```txt
LogLine canon references
conformance references
engine references
practice profiles
runtime registry material
projector definitions
examples
fixtures
standards adapters
```

Material carregado deve manter seu papel. Uma referência não é autoridade por proximidade. Um practice profile não é Foundation canon. Um overlay local não é lei universal.

### Declare

Declare cria a forma local do Lab.

Uma declaração de Lab nomeia:

```txt
lab name
purpose
operator or steward
practice profile
runtime procedures
projector set
ghost policy
receipt policy
evidence policy
proof policy
clock discipline
constraints
```

A declaração não é uma página de marca. É o contrato operacional para esta instância de Lab.

### Observe

Observe trata linguagem, sistemas, modelos, contratos, computações, simulações e decisões humanas como possível material LogLine.

Observação não iguala prova. Observação torna-se útil apenas quando é admitida, referenciada, projetada ou preservada como ghost.

### Emit

Emit cria LogLine Acts ou candidate acts.

Acts podem representar:

```txt
claims
questions
probes
runtime results
decisions
bindings
permissions
contracts
experiments
receipts
ghosts
state changes
reports
```

Material candidate permanece candidate até cruzar a fronteira de admissão declarada do Lab.

### Project

Project transforma acts em mundos legíveis.

Uma projection pode produzir:

```txt
registry
runtime view
evidence view
receipt view
ghost view
experiment view
audit view
report
standards adapter
```

Projection dá significado de domínio. Projection não é fonte semântica.

### Learn

Learn é a razão de existência do Lab.

Learning pode produzir:

```txt
new probes
new projectors
new runtimes
new ghosts
new receipts
new local rules
new upstream proposals
failures worth preserving
```

Um Lab que não preserva o que falhou em fechar não está fazendo trabalho de Lab.

## Act-Centered Operational Semantics

O produto deve expor LogLine Lab como materiais usáveis, não como vocabulário abstrato.

Cada elemento deve aparecer em pelo menos uma destas formas:

```txt
template
example
invalid example
bench
report section
projector example
conformance fixture
command or procedure
```

### LogLine Act

O ato é a unidade semântica.

Shape canônico:

```txt
who
did
this
when
confirmed_by
if_ok
if_doubt
if_not
status
```

Arquivos, tabelas, views, reports, indexes, envelopes podem apoiar o ato. Não o substituem.

### Runtime Envelope

O runtime envelope é provenance metadata ou pack/profile practice.

Responde:

```txt
which runtime
which engine
which run
which admission path
which observation path
which hashes
which prior acts
```

O ato é conteúdo. O envelope é provenance metadata ou pack/profile practice. Não é um décimo slot semântico.

### Runtime Registry

O runtime registry nomeia o que um runtime pode emitir, admitir, observar e projetar.

Impede maquinário anônimo de agir como se todos os outputs tivessem o mesmo status.

### Projection

Projection é um read model, adapter, view, report, table, API, dashboard ou mapping derivado de acts.

Projection é útil. Projection não é verdade.

### Ghost

Um ghost é ausência preservada.

Nomeia o que está faltando, por que importa, o que bloqueia, quem ou quê owns o próximo ato, e o que fecharia ou estreitaria.

Um ghost não é falha. Um ghost é o Lab recusando a mentir.

### Probe

Uma probe é uma tentativa limitada de tocar realidade, computação, evidence ou um standard.

Preserva:

```txt
claim under test
method
runtime or actor
inputs
expected observation
evidence path
doubt path
failure path
```

Uma probe pode produzir observation, evidence, ghost ou próxima probe. Não garante fechamento.

### Evidence

Evidence é observação referenciável.

Pode vir de runtime output, content hash, witness, documento assinado, model output sob limites declarados, resultado de simulação, teste de conformance, report de standard externo, trace de transação, record de performance de contrato.

Evidence não é automaticamente receipt.

### Receipt

Um receipt é fechamento escopado projetado de acts admitidos e evidence.

Um receipt declara qual escopo está fechado, qual evidence suporta fechamento, o que permanece fora do escopo, e quais ghosts permanecem.

Um receipt não deve fechar claims amplos de evidence estreita.

### Clock

O clock dá ritmo a observation, routing, probing, reporting, ghost review, state projection e receipt preparation.

Um Lab sem clock é uma pasta com ambição.

### Interval

Um interval é um campo limitado entre boundary acts.

Dentro de um interval, probes, projections, ghosts e reports operam com escopo claro.

### Standard

Um standard é referência externa que pode ser adaptada via projection.

Standards não governam semanticamente. São adaptados via LogLine Acts e projections.

### Contract

Um contrato é acordo LogLine-shaped que pode ser emitido como Act, projetado, testado e fechado.

### AI Boundary

AI é ferramenta, não autoridade.

Model output é material candidate. Não é receipt, evidence ou authority por si só.

AI pode sugerir Acts, mas não fecha verdade.

## Storage Profile-Specific

Storage é profile-specific, não canon universal.

### Online Spine (Supabase Profile)

Para o profile Supabase:

```txt
Official LogLine Acts live in the online database.
ops.logline_acts é o spine oficial.
```

### Local Buffer (Todos os Profiles)

SQLite existe como:

```txt
provisional cache
high-frequency local buffer
queue
retry state
outbox
```

SQLite não pode ser chamado de:

```txt
truth
spine
official
semantic store
source of truth
```

### Files / JSON / JSONL

Arquivos podem ser:

```txt
examples
fixtures
stdout previews
exports
debug outputs
report output
templates
```

Arquivos não podem ser:

```txt
official storage
official spine
official LogLine Act spine
official receipt storage
official ghost storage
```

## Product Surfaces

O produto tem cinco superfícies principais:

```txt
Start
Practice
Study
Build
Reference
```

### Start

Primeira experiência, instalação, initial session, getting started.

### Practice

Experiência viva do Lab: Load → Declare → Observe → Emit → Project → Learn.

### Study

Benches para estudar acts, runtimes, engines, projections, ghosts, receipts, intervals, experiments.

### Build

Ferramentas para construir com LogLine: projectors, adapters, reports, hooks.

### Reference

Documentação de referência, canon, schemas, exemplos, conformance.

## Release Standard

O produto deve ser:

```txt
Instalável
Auditável
Recoverable
Upgrade-safe
Profile-capable
```

Não deve ser:

```txt
Black box
Unrecoverable
Tied to single storage
Tied to single frontend
Tied to single runtime
```
